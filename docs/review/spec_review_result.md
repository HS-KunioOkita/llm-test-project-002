# 設計書レビュー結果

対象:
- `docs/requirements.md`
- `docs/analysis.md`
- `docs/api.md`
- `docs/design.md`
- `docs/schema.md`

レビュー観点:
- 設計上の矛盾
- 要件漏れ / 要件外追加
- 非機能 / セキュリティ / 運用保守
- 詳細設計・実装へのインプットとしての明確性

---

## 総評

全体として、要件分析を踏まえてモジュラモノリス、RBAC、監査ログ、CSV、日次バッチまで具体化されており、実装に進めるだけの骨格はあります。  
一方で、ドキュメント間の矛盾がいくつか残っており、そのまま詳細設計・実装に入ると API/権限/DB の再設計が発生する論点があります。特に、ID 設計、売上閲覧権限、監査ログ閲覧の DB 権限制御、受注明細更新仕様は着手前に確定すべきです。

---

## 指摘事項

### 1. 重大: ID 桁数設計が矛盾しており、そのままでは実装不能

- 該当:
  - `docs/schema.md` 冒頭
  - `docs/schema.md` 各テーブル定義
- 指摘:
  - `ID: 全テーブル ULID 文字列 (CHAR(26))` としつつ、例では `ord_01H...` `cus_01H...` のようなプレフィックス付き ID を前提にしています。
  - ULID 本体は 26 文字なので、`ord_` 等のプレフィックスを付けると `CHAR(26)` に収まりません。
- 影響:
  - Prisma schema、DB 定義、API レスポンス、採番処理の前提が破綻します。
- 対応案:
  - `CHAR(26)` を維持するならプレフィックスをやめる。
  - プレフィックスを維持するなら `VARCHAR(30〜32)` 等に変更し、全ドキュメントで統一する。

### 2. 重大: 売上閲覧権限が設計書内で矛盾している

- 該当:
  - `docs/design.md` 3.1 `sales` モジュール
  - `docs/design.md` 5.2 RBAC 表
  - `docs/api.md` 8. 売上集計
- 指摘:
  - `sales` モジュールの説明では「管理者・閲覧」となっていますが、RBAC 表では `admin / orderer / sales / viewer` 全ロールが売上閲覧可能になっています。
  - `api.md` には `/api/sales/daily` の権限制約が明記されていません。
- 影響:
  - 実装者がどちらを正とすべきか判断できず、権限バグやレビュー差し戻しにつながります。
- 対応案:
  - 売上閲覧の許可ロールを 1 つに確定し、`design.md` と `api.md` に同一内容を明記する。

### 3. 重大: 監査ログ閲覧 API と DB 権限制御の方針が両立していない

- 該当:
  - `docs/design.md` 6.3, 6.4
  - `docs/schema.md` 2.7
  - `docs/api.md` 9. 監査ログ
- 指摘:
  - `schema.md` では `audit_logs` は「アプリ DB ユーザーには INSERT のみ許可し、管理者ロール接続のみ SELECT 可能」とあります。
  - しかし `GET /api/audit-logs` は API サービス経由で提供する設計です。通常の API 用 DB 接続が `SELECT` 権限を持たないと実装できません。
- 影響:
  - 監査ログ閲覧 API が成立しません。接続ユーザー分離や専用 read 接続が必要です。
- 対応案:
  - 「API 用 read/write 接続を分離する」のか、「アプリ接続に `audit_logs` の SELECT を許可し、アプリ層の認可で制御する」のかを明記する。

### 4. 高: 受注明細の更新仕様が曖昧で、実装ごとに挙動が割れる

- 該当:
  - `docs/api.md` 6.4 受注更新
  - `docs/schema.md` 2.5 `order_lines`
- 指摘:
  - `PATCH /api/orders/:id` は「受注登録と同形（部分更新可）」とありますが、明細更新時のルールが未定義です。
  - 例:
    - `items` 未指定時は明細据え置きか
    - `items` 指定時は全置換か差分更新か
    - 明細削除はどう表現するか
    - `line_no` をクライアントが送るのか、サーバで再採番するのか
- 影響:
  - FE/BE で仕様解釈が割れやすく、監査ログの before/after 粒度にも影響します。
- 対応案:
  - 初版は「`items` 指定時は全置換」といった明確なルールを定め、リクエスト/レスポンス例に反映する。

### 5. 高: 要件にない機能が複数追加されており、スコープ合意が必要

- 該当:
  - `docs/api.md` 3. ユーザー
  - `docs/api.md` 8. 売上集計閲覧
  - `docs/api.md` 9. 監査ログ閲覧
  - `docs/design.md` 3.1 `users` / `audit` / `sales`
- 指摘:
  - 要件上は「管理画面」はありますが、ユーザー CRUD、パスワード再発行、監査ログ閲覧、売上集計閲覧 API までは明示されていません。
  - `analysis.md` でもユーザー管理機能は「要確認」とされていますが、本設計では確定事項として入っています。
- 影響:
  - 実装工数が増え、MVP 範囲が不明確になります。
- 対応案:
  - 「MVP 必須」「将来拡張」「要件確認待ち」に分類し直す。
  - 少なくとも `users` と `reset-password` は要件確定前提で注記すべきです。

### 6. 高: 受注・出荷の同時更新に対する排他/競合制御が未設計

- 該当:
  - `docs/api.md` 6.5, 6.6, 6.7
  - `docs/design.md` 3.2
  - `docs/schema.md` 4
- 指摘:
  - 承認、キャンセル、出荷更新は状態遷移だけ定義されていますが、同時操作時の競合制御がありません。
  - 例:
    - A が承認、B が同時にキャンセル
    - A/B が同じ受注を同時に出荷
- 影響:
  - 二重更新や不整合の原因になります。
- 対応案:
  - `WHERE status = 'draft'` を含む条件付き更新、または `updated_at`/`version` による楽観ロック方針を設計に追加する。

### 7. 中: 性能要件「2 秒以内」の測定基準が依然として曖昧

- 該当:
  - `docs/requirements.md`
  - `docs/analysis.md` 2.4, 4.4
  - `docs/design.md` 8, 10
- 指摘:
  - 要件分析では曖昧点として挙がっていましたが、設計書では測定指標が確定していません。
  - 画面応答なのか API 応答なのか、平均/P95 のどれなのか、対象 API は何かが未定義です。
- 影響:
  - 性能試験の合否判定ができません。
- 対応案:
  - 例: 「一覧/詳細/登録系の API は同時 100 ユーザー条件で P95 2 秒以内」といった受け入れ基準を明記する。

### 8. 中: 税額・端数処理ルールが要件外の仮定で固定されている

- 該当:
  - `docs/api.md` 5. 商品
  - `docs/schema.md` 2.5 `order_lines.amount`
- 指摘:
  - `unitPrice` は税抜、`taxRate` は小数、`amount = floor(quantity * unit_price * (1 + tax_rate))` とありますが、税計算・端数処理は業務ルールとして重要です。
  - `analysis.md` でも税の扱いは未定義でした。
- 影響:
  - 会計・請求との不整合が起こり得ます。
- 対応案:
  - 「税抜/税込」「行単位/伝票単位」「切り捨て/四捨五入/切り上げ」を明示し、要件確認事項として残す。

### 9. 中: 論理削除したマスタの業務キー再利用可否が不明

- 該当:
  - `docs/schema.md` 2.2 `customers`
  - `docs/schema.md` 2.3 `products`
  - `docs/api.md` 7. CSV インポート
- 指摘:
  - `customers.code` / `products.code` は一意制約ですが、論理削除後に同一コードを再利用できるかが未定義です。
  - CSV の `upsert` とも関係します。
- 影響:
  - 運用ルール次第で DB 制約が不足または過剰になります。
- 対応案:
  - 「論理削除後も再利用不可」で進めるなら現状で良い。
  - 再利用可なら部分一意制約や履歴設計の見直しが必要です。

### 10. 中: 初期パスワードを API レスポンスで返す設計はセキュリティ上の配慮が不足

- 該当:
  - `docs/api.md` 3. ユーザー `POST /api/users`
- 指摘:
  - `initialPassword` をレスポンスで平文返却する設計になっています。
- 影響:
  - API ログ、ブラウザ開発者ツール、運用者の取り扱いで漏えいリスクが上がります。
- 対応案:
  - ワンタイム初期化トークン方式、初回ログイン時パスワード変更必須、または管理者が手動設定した仮パスワードを別経路通知する方式を検討すべきです。

### 11. 中: エラーコード定義が API 仕様内で不整合

- 該当:
  - `docs/api.md` 1.3
  - `docs/api.md` 2. 認証
- 指摘:
  - 共通エラー一覧には `423 LOCKED` がありませんが、ログイン失敗時の説明では `423 LOCKED` / `ACCOUNT_LOCKED` を使用しています。
- 影響:
  - 実装時の共通エラーハンドリングがぶれます。
- 対応案:
  - 共通エラー定義に `423` を追加し、コード体系を統一する。

### 12. 中: 運用保守として必要なバックアップ/リストア、監視、障害対応設計が不足

- 該当:
  - `docs/design.md` 全体
- 指摘:
  - ログ、ヘルスチェック、バッチ再実行まではありますが、以下が未記載です。
  - DB バックアップ/リストア方針
  - 障害時の復旧手順
  - 監視対象（API 異常率、DB 接続数、バッチ失敗など）
  - 監査ログ保持 2 年後の削除運用
- 影響:
  - 本番運用開始時に運用設計を別途やり直す必要があります。
- 対応案:
  - 少なくとも運用前提として、バックアップ頻度、リストア確認頻度、最低限の監視項目を設計書に追加する。

## 確認・補足したい論点

- 「受発注管理システム」という名称に対し、発注（仕入）機能は引き続きスコープ外の理解でよいか。
- 閲覧専用ユーザーの参照範囲は全件か、自担当・自部門のみか。
- 営業担当の受注更新権限は「自分が登録した下書きのみ」で確定か。
- CSV は部分成功を許容する設計だが、業務上それでよいか。
- 売上集計は `approvedAt` 基準計上でよいか。出荷基準/売上確定基準との整合確認が必要ではないか。

---

## レビュー結論

実装着手前に最低限クローズすべき項目は以下です。

1. ID 仕様の統一
2. 売上閲覧権限の統一
3. 監査ログ閲覧の DB 権限制御方式の確定
4. 受注明細更新仕様の確定
5. 要件外機能の MVP 範囲整理
6. 競合制御方針の追加

この 6 点を解消すれば、詳細設計・実装に進める品質にかなり近づきます。
