# 要件分析: 中小卸売業向け受発注管理システム

対象: `docs/requirements.md`
想定体制: 3〜5 名チーム（FE / BE / インフラ兼任を含む）
立場: シニアテックリード兼ソフトウェアアーキテクト視点

---

## 1. 要件の要約

中小卸売業を対象とした Web 版の受発注管理システム。4 種の利用者（営業 / 受注 / 管理者 / 閲覧）が、取引先・商品マスタ管理、受注登録・承認、出荷ステータス管理、CSV 入出力、日次売上集計、監査ログ付きの操作を行う。

- 技術スタック: React + TS（FE）/ Node.js + TS（BE）/ PostgreSQL / REST API
- 運用: Docker Compose でローカル起動、外部 SaaS 禁止、テストコード必須
- 規模: 同時利用 100 名、一般操作 2 秒以内
- 基本フロー: 営業が受注登録 → 受注担当/管理者が承認 → 出荷ステータス遷移（未出荷 / 出荷済み / キャンセル） → 日次バッチで売上集計

ビジネス的には「承認」を軸にした権限分離型のトランザクショナル CRUD + マスタ管理 + バッチ集計という、オーソドックスな業務 Web アプリ。

---

## 2. 不明点・曖昧点

優先的に発注元へ確認すべき項目。

### 2.1 業務ドメイン
- **「受注」のデータモデル**: ヘッダ＋明細の想定か、単票か明示なし（仮定: 1 受注 = ヘッダ + 複数明細）
- **価格・通貨・税**: 税率、単価の上書き可否、値引き、通貨（仮定: JPY 単一、税込/税抜の扱いは未定義）
- **在庫管理の有無**: 「出荷可能」とあるが在庫チェックの要否が不明（仮定: 初版は在庫引き当てなし）
- **発注（仕入）機能**: システム名は「受発注」だが、要件には受注側しか記述がない
- **キャンセルのビジネスルール**: 承認後のみキャンセル可か、出荷済みからのキャンセル（返品扱い）可か不明
- **出荷ステータス遷移**: 「未出荷 → 出荷済み」以外に、キャンセルへの遷移条件、逆行不可かが未定義
- **承認の単位**: 受注単位 / 明細単位 / 金額閾値による多段承認か不明（仮定: 受注単位の単段承認）

### 2.2 ユーザー・権限
- **閲覧専用ユーザー**の閲覧範囲（全社 / 担当分のみ）が不明
- **営業担当の担当取引先スコープ**（自分の担当分だけ登録可能か全社可か）不明
- **管理者**の範囲（ユーザー管理・マスタ削除・論理/物理削除権限）が未定義
- **ユーザー管理機能**自体（登録・パスワード初期化・ロール付与）がスコープに明記されていない

### 2.3 CSV・バッチ
- **CSV フォーマット**（文字コード SJIS/UTF-8、区切り、ヘッダ有無、エラー時の扱い）未定義
- **CSV のサイズ上限**（1 万行？10 万行？）未定義。2 秒ルールとの整合が取れない可能性
- **日次バッチの実行時刻・失敗時の再実行/通知**未定義
- **売上集計の粒度**（取引先別 / 商品別 / 日別 / 月別、確定済みのみか）未定義
- **CSV エクスポート**は要件に明示がない（インポートのみ記載）

### 2.4 非機能
- **2 秒以内**の対象（画面遷移？API 応答？95 パーセンタイル？）未定義
- **可用性目標（SLA）**、RPO/RTO、バックアップ要件が未記載
- **データ保持期間**（監査ログ、受注履歴）未定義

## 3. 要件上の矛盾・不足

- **システム名と機能範囲の不一致**: 「受発注」を名乗るが、発注（仕入）側機能がスコープにない。
- **CSV 入出力とあるが、要件文では「インポート」のみ**。エクスポートの有無が曖昧。
- **出荷ステータスの遷移グラフ未定義**。単なる列挙しかなく、状態遷移の妥当性検証ができない。
- **認証・認可の要件そのものが欠落**。「ユーザー種別」はあるが、ログイン方式・パスワードポリシー・セッション管理の指定なし。
- **監査ログの要件粒度が不足**: 誰が / いつ / 何を / 旧値・新値 / 理由、の項目指定なし。保存期間・改ざん対策（追記専用）の指針なし。
- **個人情報・取引情報の取扱い**についての記述なし（卸売で取引先担当者情報を扱う可能性が高い）。
- **エラーハンドリング / バリデーション規約**の記述なし。
- **環境（開発/ステージング/本番）やデプロイ先**の記述なし。Docker Compose だけで本番運用する想定か不明。

## 4. 非機能・運用要件として追加で考慮すべき点

要件に明記されていないが、業務 Web アプリとして実質的に必要になる項目。

### 4.1 セキュリティ
- 認証（仮定: ID/パスワード + セッション or JWT）、パスワードポリシー、ロックアウト
- 認可（RBAC）の明示的な実装と、画面・API 両層での強制
- CSRF / XSS / SQLi 対策、セキュリティヘッダ、監査ログへの操作者・IP 記録
- 秘匿情報（DB 接続情報、JWT 鍵）の環境変数管理と `.env.example`

### 4.2 運用
- ロギング方針（構造化ログ、相関 ID、ログレベル）
- ヘルスチェックエンドポイント（`/healthz`, `/readyz`）
- バッチの再実行性（冪等性）・失敗通知・実行履歴
- バックアップ・リストア手順（少なくともドキュメント上）
- マイグレーション運用（Prisma Migrate や node-pg-migrate）

### 4.3 品質
- CI（Lint / 型チェック / テスト自動実行）
- テストの層別方針（単体 / 結合 / E2E）とカバレッジ基準（目安: BE ロジック 70%+）
- ER 図・API 仕様（OpenAPI 推奨）の自動生成 or 更新プロセス

### 4.4 性能
- 受注一覧の検索用インデックス設計（取引先、ステータス、期間）
- N+1 回避、ページング仕様（オフセット or カーソル）
- 2 秒 SLA の測定基準を P95 応答時間として定義

### 4.5 アクセシビリティ・ブラウザサポート
- 対象ブラウザ、最低限の a11y（キーボード操作・コントラスト）

## 5. 実装上の主要リスク

優先度つきで列挙。

| # | リスク | 影響 | 対策の方向性 |
|---|--------|------|----------|
| R1 | 権限モデルが曖昧なまま実装に入ると、後工程で全 API 改修になる | 大 | 設計初期にロール×リソース×操作の権限マトリクスを確定 |
| R2 | 受注の状態遷移ルールの後出し | 大 | ステートマシンを先に図で固め、テストで網羅 |
| R3 | CSV インポートの文字コード・件数・エラー行扱いで泥沼化 | 中〜大 | 初版は UTF-8 限定・件数上限・ドライラン方針で合意 |
| R4 | 監査ログの設計が甘いと、後で業務監査に耐えない | 中 | ログは追記専用テーブル + DB トリガ or アプリ層集約で最初から担保 |
| R5 | 100 名同時利用 / 2 秒 SLA に対する性能検証の欠落 | 中 | 受注一覧・CSV・バッチに対する負荷試験観点を最初から定義 |
| R6 | 日次バッチが冪等でないと、失敗再実行で集計が二重計上 | 中 | 集計テーブルを `(対象日, キー)` で UPSERT、もしくは日次削除→再投入 |
| R7 | 在庫連携の後付け要求 | 中 | 受注明細と在庫の関係を考慮し、受注ドメインの責務を広げすぎない設計 |
| R8 | 3〜5 名の小規模チームで過剰な分割（マイクロサービス化など）に走ると破綻 | 中 | モノリシックな単一 BE + SPA フロントで開始 |
| R9 | ORM 選定ミスで生 SQL 多用地獄 or 抽象化漏れ | 中 | Prisma を軸に、複雑集計のみ生 SQL を許容 |

## 6. 推奨アーキテクチャの方向性

### 6.1 基本方針
- **モジュラモノリス**で開始。BE は単一 Node.js サービス、ドメインごとにモジュール分割（`orders` / `customers` / `products` / `shipments` / `audit` / `auth` / `batch`）。
- FE は React + TypeScript + Vite の SPA。状態管理は React Query を中心に、グローバル状態は最小限。
- DB は PostgreSQL 単一インスタンス。スキーマは単一、監査ログのみ追記専用テーブルで分離。
- API は REST + OpenAPI。認証はセッションベース（Cookie, SameSite=Lax, HttpOnly）を推奨（仮定: 社内利用であり SPA 同一ドメイン配信できる前提）。

### 6.2 レイヤ構成（BE）
```
  Controller (HTTP/REST)
    └─ UseCase / Service (トランザクション境界)
         └─ Domain (エンティティ、状態遷移ルール)
              └─ Repository (Prisma)
```
- 状態遷移（受注承認・出荷ステータス）はドメイン層に閉じ込め、ユニットテスト容易性を確保。
- 横断関心事（認証・認可・監査ログ・相関 ID）はミドルウェア/デコレータで統一。

### 6.3 技術選定の具体案（仮定込み）
- BE: Fastify or NestJS（NestJS は中規模以上で DI/構造化のメリット、3〜5 人ならどちらも可）
- ORM: Prisma（マイグレーション、型安全性、学習コストのバランスが良い）
- バリデーション: Zod（FE/BE 共有可）
- テスト: Vitest（FE/BE 統一しやすい）、BE 統合テストは testcontainers で PostgreSQL 起動
- バッチ: 初版は単純に `node dist/batch/daily-summary.js` を cron/Docker で実行。専用 DSL は不要。
- CSV: `papaparse`（FE プレビュー） + `csv-parse`（BE）
- ロギング: pino（構造化）

### 6.4 デプロイ
- ローカル: Docker Compose（`web` / `api` / `db`）
- 本番想定は要確認だが、仮定としてコンテナ実行可能な任意の環境（ECS / GCE / オンプレ Docker）で動作する構成とする。

## 7. 実装優先順位（MVP 観点）

「承認 → 出荷」という業務コア動線を最短で通すことを最優先とする。

### Phase 0: 基盤（1〜2 週間）
- リポジトリ・モノレポ構成、CI（lint/型/テスト）
- Docker Compose（api / web / db）
- 認証基盤（ログイン・セッション・ロール）と RBAC ミドルウェア
- 監査ログ基盤（追記テーブル + 共通ラッパ）
- マイグレーション運用・seed データ

### Phase 1: MVP（業務コア動線、3〜4 週間）
1. 取引先 CRUD（検索・ページング込み）
2. 商品 CRUD
3. 受注登録（ヘッダ＋明細）
4. 受注承認（権限分離）
5. 出荷ステータス更新（状態遷移のガード付き）
6. 受注一覧（検索・ページング）
7. 最低限の画面（SPA）と OpenAPI ドキュメント

この時点でエンドツーエンドのデモが可能であることを MVP 完了基準とする。

### Phase 2: 周辺機能（2〜3 週間）
8. CSV インポート（取引先・商品、UTF-8 固定、ドライラン、エラー行レポート）
9. 日次売上集計バッチ（冪等な UPSERT）
10. 管理画面（ユーザー管理・ロール付与）※要件確認のうえ
11. 監査ログ閲覧画面

### Phase 3: 品質・運用（1〜2 週間）
12. 性能試験（受注一覧・CSV・バッチ）と 2 秒 SLA の検証
13. E2E テスト（Playwright 最低限）
14. ドキュメント整備（README / ER 図 / OpenAPI / 運用手順）

### 意図的に「後回し」または「スコープ外」候補
- CSV エクスポート（要件に明記がないため、Phase 2 で要確認）
- 発注（仕入）機能
- 在庫管理
- 多段承認・金額閾値承認
- 通知（メール・Slack 等）
- 多言語・多通貨
